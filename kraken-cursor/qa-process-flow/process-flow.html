kraken-cursor  ·  QA Bot Factory

QA Process Flow

Two commands to run QA across any client C# codebase. Auto-discovers projects, runs checks, streams progress live.

One-Time Setup
01
Clone QA Bots repo
git clone https://github.com/gocrisp-proservices/QA-Bots <path-to-QA-Bots>
02
Install kraken-cursor skills in Cursor
Add the SkillsOfTheKraken kraken-cursor/ skills to your Cursor skills path.
03
Note your QA Bots repo path
On first run of kraken-cursor-qa-init, you'll be asked once for this path. It's saved to ~/.cursor/.kraken-cursor/qa-bots-path automatically — never asked again.
Per-Client Workflow
Start
Open client repo in Cursor
Open the client's codebase as your working directory. The .sln file must be reachable from this folder.
↓
First time only
kraken-cursor-qa-init
Auto-detects .sln, confirms client name from folder, scaffolds QA Bots/clients/<name>/ with config and skills. Zero questions (usually).
↓
Run any time
kraken-cursor-qa-run
Parses the .sln, scans each project for Dapper / web endpoints / Snowflake, runs applicable checks, writes results.json.
↓
Auto-launched
Terminal Watcher
New PowerShell window streams live project status as checks complete.
Auto-launched
Browser Dashboard
Dark-theme HTML report with per-project cards, status badges, and expandable issue details. Refreshes every 3 seconds until complete.
↓
Fix & repeat
Fix issues in client code, re-run
Address issues flagged in the dashboard. Run kraken-cursor-qa-run again — no re-init needed.
Commands Reference
kraken-cursor-qa-init
Scaffold QA workspace for current client repo. Detects .sln, creates config, copies skills.
WHEN: first time per client
kraken-cursor-qa-run
Run full QA analysis. Parses solution, detects checks, launches watcher + dashboard.
WHEN: any time you want to QA
kraken-cursor-qa-dapper <file.cs>
Validate a single C# file for Dapper issues: SQL injection, nullability, mapping parity, connection leaks.
WHEN: called automatically by qa-run, or manually
kraken-cursor-qa-web-smoke <url>
Smoke test a staging URL via browser automation. Checks console errors, HTTP status, layout.
WHEN: called automatically by qa-run, or manually
kraken-cursor-qa-openaccess <projectPath>
Test OpenAccess ASCX controls. Finds parent pages in local CKB DB, logs in as ckbadmin, navigates each page, screenshots.
WHEN: called automatically by qa-run, or manually per OA project

WHEN: need a reminder
What To Expect
Multiple .sln qa-init lists them and asks which to use.
No .sln found qa-init asks for the path directly.
Web project, no staging URL qa-run asks once, saves to qa-config.json for future runs.
Snowflake detected Project marked Manual Review — automated checks skipped, flagged in dashboard.
Test projects (.Tests) Automatically skipped — not production code.
Client name collision qa-init appends -2, -3, etc. and notifies you.
OA control, no page qa-openaccess flags as warning — ASCX control has no navigable parent page in ix_web_page. Usually means it's a popup/panel triggered via grid right-click.
OA login fails Verify OpenAccess is running locally at https://<hostname>/ikb and ckbadmin account is active.
