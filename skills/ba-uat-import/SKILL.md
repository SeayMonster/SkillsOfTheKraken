# BA — Import UAT Excel into Notion Database

<context>

## What this skill does

Reads a UAT Excel file the BA has uploaded to the Academy Notion page, maps the columns to the Notion UAT database schema, previews the import with the BA, then creates the rows in the UAT database.

---

## The rule

> **Notion is the client source of truth. GitHub is the dev source of truth.**
> BA files (Excel, Word, etc.) always go to Notion — never committed to the GitHub repo.

---

</context>

<task>

## Step 1 — Load config

Read `docs/ba-config.json`.

- If `notion.uat_database_id` still contains `REPLACE_WITH_`, STOP:
  > "The Notion UAT database hasn't been set up yet. Ask your dev to create the UAT database in Notion and fill in `docs/ba-config.json`."

Store:
- `UAT_DB_ID` = `notion.uat_database_id`
- `ACADEMY_PAGE_ID` = `notion.academy_page_id`

---

## Step 2 — Get the Excel file path

Ask the BA for the path to their Excel file:
> "What's the path to your UAT Excel file? You can drag the file into this chat or paste the full path (e.g. `C:\Users\you\Documents\Academy_UAT.xlsx`)."

The file stays on the BA's machine — it does not need to be uploaded anywhere. PowerShell reads it locally and the data goes straight into Notion.

---

## Step 3 — Extract data from Excel using PowerShell

Use the PowerShell tool to read the Excel file. Try COM automation first (requires Excel installed), fall back to ImportExcel module.

**Attempt 1 — COM automation (Excel installed):**
```powershell
$path = "C:\path\to\file.xlsx"   # use the path the BA provided
$xl  = New-Object -ComObject Excel.Application
$xl.Visible = $false
$wb  = $xl.Workbooks.Open($path)

$results = @()
foreach ($sheet in $wb.Sheets) {
    $ws       = $sheet
    $lastRow  = $ws.UsedRange.Rows.Count
    $lastCol  = $ws.UsedRange.Columns.Count
    # Row 1 = headers
    $headers  = 1..$lastCol | ForEach-Object { $ws.Cells.Item(1, $_).Text }
    for ($r = 2; $r -le $lastRow; $r++) {
        $row = [ordered]@{ _sheet = $ws.Name }
        for ($c = 1; $c -le $lastCol; $c++) {
            $row[$headers[$c-1]] = $ws.Cells.Item($r, $c).Text
        }
        $results += [PSCustomObject]$row
    }
}
$wb.Close($false)
$xl.Quit()
[System.Runtime.Interopservices.Marshal]::ReleaseComObject($xl) | Out-Null
$results | ConvertTo-Json -Depth 3
```

**Attempt 2 — ImportExcel module (no Excel required):**
```powershell
if (-not (Get-Module -ListAvailable -Name ImportExcel)) {
    Install-Module ImportExcel -Scope CurrentUser -Force
}
Import-Module ImportExcel
Import-Excel "C:\path\to\file.xlsx" -WorksheetName * | ConvertTo-Json -Depth 3
```

If both fail, tell the BA:
> "I couldn't read the Excel file. Make sure Excel is installed or let me install the ImportExcel module (one-time, takes ~30 seconds). Reply 'yes' to install it."

**Reading multiple sheets:** Each sheet in the workbook is treated as a separate feature area or test group. The `_sheet` property carries the sheet name through for mapping in Step 4.

---

## Step 3.5 — Sync guard (only if file came from ba-uat-pull)

After extracting Excel data, check whether the file contains a `_SyncMeta` sheet and a `_PullInfo` sheet.

**If neither sheet exists:** This is a hand-crafted Excel file, not a pulled file. Skip the sync guard entirely and proceed to Step 4.

**If the sheets exist:** The file was generated by `/ba-uat-pull`. Run the conflict check:

```powershell
$meta     = Import-Excel $path -WorksheetName '_SyncMeta'
$pullInfo = Import-Excel $path -WorksheetName '_PullInfo'
$pullTime = [datetime]$pullInfo.PullTimestamp

# Build a lookup: NotionUrl → LastEditedTime at pull time
$snapshots = @{}
$meta | ForEach-Object { $snapshots[$_.NotionUrl] = [datetime]$_.LastEditedTime }
$snapshots | ConvertTo-Json
```

Use `notion-query-data-sources` to re-fetch the current version of every row whose `NotionUrl` appears in the snapshot. For each row:

- If `current.lastEditedTime` > `snapshot.LastEditedTime` **AND** the Excel row content differs from the snapshot → **CONFLICT**: Notion was edited after the BA pulled AND the BA also changed it locally.
- If `current.lastEditedTime` > `snapshot.LastEditedTime` **AND** the Excel row matches the snapshot → Notion changed, BA did not touch it → treat as **Notion-wins** (no overwrite needed, skip this row).
- Otherwise → proceed normally through the diff in Step 5.

Store conflicts as `CONFLICTS` list. Store Notion-wins rows as `NOTION_ONLY_CHANGES` (these are excluded from the update step — Notion already has the right version).

If `CONFLICTS` is non-empty, **stop and show the BA before doing anything else:**

```
⛔ Sync conflict detected — Notion was edited after your pull.

These rows were changed in BOTH your Excel AND Notion:

  Row: [Test Case name]
    Your Excel version:  [Steps excerpt]
    Current Notion:      [Steps excerpt]
    Notion edited at:    [timestamp]

  Row: [Test Case name]
    ...

For each conflicted row, tell me:
  - "keep mine"  → your Excel version overwrites Notion
  - "keep theirs" → your Excel changes for this row are discarded
  - "skip" → leave this row unchanged, I'll fix it manually in Notion
```

Wait for the BA's decision on **every conflicted row** before continuing. Only proceed to Step 4 after all conflicts are resolved.

---

## Step 4 — Map columns to Notion UAT schema

The Notion UAT database expects these fields:

| Notion field | Required | Likely Excel column names |
|---|---|---|
| Test Case | Yes | Test Case, Test Name, Scenario, Title |
| Feature Area | Yes | Feature, Area, Module, Function |
| Steps | Yes | Steps, Test Steps, Procedure |
| Expected Result | Yes | Expected Result, Expected, Pass Criteria |
| Notes | No | Notes, Comments, Prerequisites, Data Required |
| Manual Status | No | Status, Result, Pass/Fail (default: Not Run) |

Ask the BA to confirm the column mapping if any column names are ambiguous:
> "I found these columns in your file: [list]. I'm mapping them as follows — does this look right? [show mapping table]"

If a required field has no matching column, ask the BA which column to use.

---

## Step 5 — Diff against existing Notion rows (sync, not blind import)

Before writing anything, fetch existing rows from the UAT database for the same Feature Area(s) found in the Excel.

Use `notion-query-data-sources` to get all existing rows matching those Feature Areas.

Exclude any rows flagged as `NOTION_ONLY_CHANGES` from this diff — those rows were updated in Notion after the pull but the BA didn't touch them, so Notion already has the correct version. Do not overwrite them.

Build three lists by comparing **Test Case name** (case-insensitive) as the unique key:

| List | Definition |
|---|---|
| **New** | In Excel, not in Notion → will be created |
| **Updated** | In both, but Steps or Expected Result differ → will be updated |
| **Orphaned** | In Notion, not in Excel → candidate for deletion |

> **Never auto-delete.** Always present orphans to the BA and wait for explicit confirmation before removing anything.

---

## Step 6 — Present the sync plan

Show the BA a clear summary before touching anything:

```
Sync plan for [Feature Area]:

  ✅ New (will be created):      [N] test cases
     - [Test Case name]
     - [Test Case name]

  ✏️  Updated (content changed): [N] test cases
     - [Test Case name]

  ⚠️  Orphaned (in Notion, not in your Excel): [N] test cases
     - [Test Case name]
     - [Test Case name]

  unchanged:                     [N] test cases (no action needed)
```

Then ask:
> "For the orphaned rows — do you want me to delete them from Notion, or leave them as-is?"

Wait for BA's answer on orphans before proceeding. New and updated rows can be confirmed together.

---

## Step 7 — Apply the sync

On BA confirmation:

**New rows** — `notion-create-pages` with all fields, Manual Status = Not Run.

**Updated rows** — `notion-update-page` for the matching Notion page ID. Only update: Steps, Expected Result, Notes. Never touch:
- `Automated Status` — Playwright owns this
- `Automated Run Date` — Playwright owns this
- `Final Sign-off` — BA owns this; a content update doesn't reset their sign-off
- `Sign-off By` — same

**Orphaned rows (if BA said delete)** — delete each page. If the orphaned row has `Final Sign-off = true`, warn the BA first:
> "⚠️ '[Test Case name]' was already signed off. Are you sure you want to delete it?"
Require explicit yes before deleting a signed-off row.

Report any failures row-by-row and continue with the rest.

---

## Step 8 — Confirm and clean up

After sync:
> "Sync complete:
>  - [N] created
>  - [N] updated
>  - [N] deleted
>  - [N] unchanged
>
> You can delete the uploaded Excel/CSV from the Academy Notion page now — the database rows are your source of truth going forward."

## Excel template tips (share with BA)

For smooth imports the BA's Excel should follow these conventions:

| Column | Tips |
|---|---|
| Test Case | Short name, unique per row — becomes the Notion row title |
| Feature Area | Use consistent values: POGSplit, LifecycleManagement, FloorPlanning, SpacePlanning |
| Steps | Numbered steps — each step on its own line or separated by semicolons |
| Expected Result | One clear statement of what "pass" looks like |
| Notes | Optional — prerequisites, required DBKeys, known constraints |
| Status | Optional — leave blank to default to "Not Run" |

**Sheet names matter:** Each worksheet name is used as the Feature Area if no Feature Area column exists. Naming sheets "POGSplit", "LifecycleManagement" etc. means the BA doesn't need a Feature Area column at all — the skill infers it from the sheet.

**No special format required:** Column names don't need to match exactly — the skill maps them in Step 4. Rough names like "Test Name", "Steps to Reproduce", "Pass Criteria" all work.
